# Write Notes Skill

根据本仓库中的 Jupyter notebook 撰写对应的学习笔记，输出到 `notes/` 目录。风格对齐 `notes/01-micrograd.md`（黄金模板）。

---

## 工作流程

### 1. 完整阅读 Notebook

逐 cell 阅读，不跳：

- **Markdown cell 是用户的理解** — 用自己的话解释概念，直接反映他认为重要的东西。笔记必须体现这些理解。
- **代码注释中的标记** — `# 重点`、`# 本质`、`# 请务必写到笔记里` 这类标注是必须覆盖的内容。
- **Tensor shape 标注** — 用户习惯在注释里跟踪 shape（如 `# (32, 3, 2)`），笔记中保留数据流。
- **承上启下的过渡语** — "下面开始..."、"但是..."、"这里..." 揭示了叙事逻辑。

### 2. 提炼故事弧线

每课都有一条主线，不是知识点罗列：

> 问题（动机） → 尝试（实现） → 发现（踩坑） → 修复（改进） → 结果 + 对比

笔记章节沿着这条线组织，让读者能跟着思维走。

### 3. 写作

**文件头固定格式：**
```markdown
# Lecture X — 课程名

## 课程仓库：[repo-name](url)

> 日期：YYYY-MM-DD
```

**必须做：**
- 中文段落叙事，连续行文
- 代码块嵌在段落之间，代码后紧跟解释"为什么这样写"
- bug/坑 展开为完整链条：问题场景 → 原因 → 修复 → 本质
- 写出关键数值：loss 值、参数数量、训练轮数、shape 变化
- 末尾放阶段对比表（对比维度：框架、输入、输出、损失、优化...）
- 每节用 `## 一、## 二、...` 编号

**禁止做：**
- 禁止碎片的 `###` 三层小标题（bug 分析的"问题/原因/修复"除外）
- 禁止教科书式百科全书解释——是学习笔记，不是参考文档
- 禁止忽略用户自己的代码注释和 markdown 见解
- 禁止无代码支撑的空泛描述

### 4. 自检清单

写完逐项确认：

- [ ] 这课解决什么问题？（开头动机）
- [ ] 数据怎么构造？（shape 变化）
- [ ] 模型结构？（每层输入输出 shape）
- [ ] 损失函数？（为什么选它，和上一课的区别）
- [ ] 训练细节？（轮数、学习率、batch 大小、参数数量）
- [ ] 用户标注的"重点"有没有遗漏？
- [ ] 有没有踩坑记录？
- [ ] 末尾有没有和上一课的对比？
- [ ] 课程仓库链接和日期？

### 5. 交叉验证

打开 `notes/01-micrograd.md` 对照：
- 段落密度是否接近？
- 代码块和文字的配比是否接近？
- 小标题使用频率是否接近？（01 用了 4 个 ###，都在第四节 bug 分析中）

## 输出

直接写入 `notes/XX-lesson-name.md`，README 进度更新等 skill 调用结束后用户会自行安排。

## 常见错误

1. **写成参考文档** — `###` 碎标题 + 列表 = API 手册。用段落叙事代替。
2. **漏掉用户注释** — 代码注释里的"重点"必须覆盖。
3. **没有对比** — 每课末尾都要和上一课对比。
4. **数值缺失** — 没有 loss 变化、参数数量的笔记缺乏质感。
5. **自行扩展** — 用户 notebook 里没写的不要加。笔记是提炼，不是扩充。
